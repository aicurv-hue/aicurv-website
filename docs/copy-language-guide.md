## Words To Use

| Use | Avoid |
|-----|-------|
| Working software | Prototype / MVP (implies not real) |
| Ship / Deploy | Launch (overused) |
| Sprint | Project / Engagement |
| Build cycle | Development process |
| Validate | Test (too passive) |
| Senior engineers | Developers / Coders |
| AI-augmented | AI-powered (buzzword, triggers skepticism) |
| Scope creep | Changes (too soft) |
| Fixed sprint | Retainer / Hours |
| Production-ready | Enterprise-grade (vague) |
| Data / Feedback | Insights (corporate) |
| Fail fast | Failure (negative framing) |

---

## Words To Avoid (and why)

| Word | Why |
|------|-----|
| **Streamline** | Vague, meaningless |
| **Optimize** | Overused, says nothing specific |
| **Innovative** | Everyone says this |
| **Cutting edge** | Developer audience finds it a credibility killer |
| **State of the art** | Same as above |
| **AI-powered** | Triggers Builder.ai association — explain what AI does instead |
| **Seamless** | Cliché |
| **Leverage** (as verb) | Corporate jargon |
| **Utilize** | Use "use" |
| **Facilitate** | Use "help" |
| **Synergy** | Never |
| Exclamation points | Remove them always |

## Headline Formulas That Work

### Speed + Specificity
> "Your MVP, shipped in 2 weeks. Not a prototype. Production software."

### Anti-agency contrast
> "Most agencies start with a Gantt chart. We start with working code."

### Philosophy contrast (our format)
> "Most agencies ask: [old question]"
> "We ask: [new question]"

### Urgency + loss aversion
> "Your competitors aren't waiting. Neither should you."

### Outcome over feature
> "Know if your idea works before you commit everything to it."

## CTA Copy Rules

| Type | Formula | Example |
|------|---------|---------|
| Primary action | [Verb] + [what they get] | Ship Your First Sprint |
| Plan CTA | [Verb] + [activity] | Start Automating / Start Building |
| Enterprise CTA | Low commitment | Let's Talk |
| Secondary CTA | Curiosity-led | See How It Works |

**Never use:** Submit, Sign Up, Learn More, Click Here, Get Started, Select Plan

## FAQ — Approved Questions & Tone

All FAQ answers follow this pattern:
1. Validate the concern (don't dismiss it)
2. Explain what we actually do
3. Land on the reassuring outcome

Example:
> **Q: Is AI-generated code actually production-ready?**
> *"This is the right question to ask."* [validates] *"AI accelerates our build process — it doesn't run unsupervised."* [explains] *"Clean architecture, documented code, no lock-in."* [reassures]
